# Техническое задание: Модуль автономной генерации поэзии «Завод Навоза 2.0» (Rev. 2)

## 1. Общее описание
Автономный модуль на Python для генерации экзистенциальных стихов в стиле «классика $\times$ абсурд» с центральной темой «навоза/бетона». Модуль полностью независим от OpenClaw и взаимодействует с LLM напрямую через API.

## 2. Функциональные требования

### 2.1. Генерация контента (`generator.py`)
- **API Взаимодействие:** Использование `httpx` для асинхронных/синхронных запросов.
- **Отказоустойчивость:** 
    - Реализация таймаутов и стратегии ретраев (Exponential Backoff) при ошибках `429` (Rate Limit) и `5xx`.
    - Логирование всех ошибок API в `data/error.log`.
- **Промпт-контракт:** 
    - Жесткое требование к модели возвращать ответ в формате: `<json> {"motifs": [], "antonyms": [], "title": "...", "poem": "..."} </json>`.
    - Наличие парсера с fallback-логикой на случай некорректного вывода модели.

### 2.2. Хранение данных (`recorder.py`)
- **Атомарная запись:** Запись через временный файл с последующим `os.replace` для предотвращения порчи данных при сбоях.
- **Конкурентный доступ:** Использование file lock (`fcntl` или `filelock`) для предотвращения повреждения `records.json` при одновременном запуске нескольких копий модуля.
- **Изоляция:** Валидация всех путей через `os.path.commonpath()` относительно корня модуля для предотвращения записи вне директории `poet_module`.
- **Версионирование:** Монотонный инкремент версии внутри атомарной транзакции чтения-записи.

### 2.3. Уведомления (`notifier.py`)
- **Реализация:** Вывод в `stdout` с пометкой `[TELEGRAM STUB]`. Архитектура должна позволять замену на реальный API-клиент без правки остального кода.

### 2.4. Координация (`main.py`)
- **Жизненный цикл:** `Валидация конфига` $\rightarrow$ `Генерация (с обработкой ошибок)` $\rightarrow$ `Атомарная запись` $\rightarrow$ `Уведомление`.
- **Логирование:** Использование стандартного модуля `logging` (вывод в консоль и файл `data/app.log`).

## 3. Технические требования

### 3.1. Стек и Безопасность
- **Язык:** Python 3.10+.
- **Зависимости:** `httpx`, `pydantic-settings` (для валидации `.env`).
- **Секреты:** Хранение API-ключа в файле `.env` (включен в `.gitignore`).
- **Контроль версий:** Git.

### 3.2. Конфигурация
- `.env`: `API_KEY`, `API_ENDPOINT`, `MODEL_NAME`.
- `requirements.txt`: Список всех зависимостей.

## 4. Структура проекта
```text
poet_module/
├── .env                 # Секреты (НЕ коммитить)
├── .gitignore           # Исключение .env, __pycache__, data/
├── requirements.txt     # Зависимости
├── TZ.md                # Данный документ
├── data/
│   ├── records.json     # Архив стихов (атомарный)
│   └── error.log        # Логи ошибок
└── src/
    ├── generator.py     # API LLM + Парсинг JSON
    ├── recorder.py      # File Lock + Atomic Write
    ├── notifier.py      # Output/Telegram Stub
    └── main.py         # Orchestrator
```

## 5. Критерии приемки
1. **Безопасность:** API-ключ не попадает в Git.
2. **Целостность:** При принудительном завершении процесса (`kill -9`) во время записи, файл `records.json` остается валидным.
3. **Изоляция:** Любая попытка записи вне папки модуля вызывает `PermissionError` или аналогичный.
4. **Монотонность:** Каждая новая запись имеет строго следующий за предыдущей номер версии.
5. **Валидация:** Скрипт отказывается запускаться, если в `.env` отсутствуют обязательные поля.
